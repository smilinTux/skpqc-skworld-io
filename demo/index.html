The hybrid KEM, live in your browser

The pure-Rust sk-pqc core, compiled to WebAssembly — running a full X25519 + ML-KEM-768 key-encapsulation round-trip locally, in this tab. No server. No upload. Your browser does the lattice math.

⚠️
Runs locally in your browser. Every byte below — keygen, encapsulation, decapsulation — is computed in WebAssembly on your machine; nothing is sent anywhere. Randomness comes from your browser's crypto.getRandomValues. This is a hybrid scheme: the derived secret holds as long as either the classical X25519 leg or the ML-KEM-768 (FIPS 203) leg holds. It is post-quantum, never "quantum-proof", "quantum-safe", or "unbreakable". Experimental & unaudited.

Hybrid KEM round-trip — Alice ⇄ Bob, in WASM

Alice generates a hybrid keypair. Bob encapsulates a fresh shared secret to her public key. Alice decapsulates it. Both sides must derive the identical 32-byte secret — then a wrong-key negative control, and a DM-ratchet message-key derivation.

What just ran — the honest version

What this demo proves

  • The same pure-Rust core on PyPI / pub.dev / crates.io compiles unchanged to wasm32 and runs in a browser
  • A real hybrid KEM: X25519 (RFC 7748) + ML-KEM-768 (FIPS 203), combined with HKDF-SHA256 — concatenate-then-KDF, never XOR
  • Round-trip agreement: encapsulator and decapsulator independently derive the same 32-byte secret
  • Implicit rejection: a wrong private key yields a non-matching secret, not a usable one
  • The DM epoch ratchet derives deterministic, index-separated per-message keys off one epoch secret

What it does not claim

  • Not "quantum-proof" — the defensible word is post-quantum; hybrid = secure if either leg holds
  • Not authentication — this is a KEM; it does not prove who Bob is (signatures are future work, FIPS 204)
  • Not audited — research-grade wiring over vetted primitives (ml-kem, x25519-dalek); no third-party review yet
  • Not a benchmark — a browser/WASM run is illustrative, not a timing claim
  • Not endpoint security — it protects a key exchange, not a compromised device or a bad RNG

The same thing, in your stack

This is exactly the API the demo calls — the WASM exports map one-to-one onto the published packages. Same construction, same wire bytes, three ecosystems.

// the WASM glue this page imports (pkg/sk_pqc_wasm.js)
import init, { hybrid_keypair, hybrid_encap, hybrid_decap, suite_id }
  from './pkg/sk_pqc_wasm.js';
await init();
const alice = hybrid_keypair();          // 1216-byte pub / 2432-byte priv
const enc   = hybrid_encap(alice.public_key); // enc.ciphertext (1120), enc.shared_secret (32)
const ss    = hybrid_decap(enc.ciphertext, alice.private_key); // === enc.shared_secret
# Python — pip install sk-pqc
import sk_pqc
pk, sk = sk_pqc.generate_keypair()
ct, ss = sk_pqc.encapsulate(pk)
ss2    = sk_pqc.decapsulate(sk, ct)        # ss == ss2

Build the WASM yourself from sk-pqc-rs: cd wasm && wasm-pack build --release --target web --out-dir pkg.
